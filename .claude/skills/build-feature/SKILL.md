# 機能実装ワークフロー

$ARGUMENTS で指定された機能を、以下のワークフローで**最後まで**実装する。

## 進め方の原則

- **完走する**: TODO(human) によるコード委譲は行わない。ユーザーはコーディングせず、PR の Files changed と解説ドシエでレビューする
- 設計判断が分かれる場面（データ構造・認可方式など）は、着手前に AskUserQuestion で選択肢＋推奨を提示して決める。要件から自明なら自分で決めて進め、決定理由を最終報告に含める
- 既存の設計パターンと整合させる：三層競合対策（部分ユニーク index＋バリデーション＋rescue_from）、二層ユニーク制約、Authentication concern のクローズ既定パターン、スコープのモデル配置

## 手順

1. **要件確認**: `requirements/requirements.md` の該当する発展要件を読み、受け入れ条件を箇条書きに整理してから着手する
2. **ブランチ**: main を最新化（`git pull origin main`）し、`feature/<機能名>` を作成
3. **TDD**: テストを先に書く。実装前に `rails test` を実行し、**意図した理由・意図した箇所で Red になっているかを失敗メッセージで目視確認**してから実装に進む（CLAUDE.md のルール）。実装がなくても通るテスト＝偽陽性に注意
4. **実装**: Green になるまで。マイグレーションには DB 層の制約（NOT NULL・ユニーク index）も検討して入れる
5. **検証**: `rails test` 全通過を確認し、verify スキルで実際の画面フローを動かして確認する
6. **PR 作成**: 何を・どう・なぜ実装したか、設計判断、テスト結果を本文にまとめて `gh pr create`
7. **ドシエ生成**: Skill ツールで `feature-dossier` を起動し（引数: この機能の PR 番号）、解説アーティファクトを生成する
8. **最終報告**: PR の URL・ドシエの URL・自分で下した設計判断とその理由を最終メッセージにまとめる
