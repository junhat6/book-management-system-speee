# 実装解説ドシエの生成

$ARGUMENTS で指定された機能について、ユーザーがレビュー会で自分の言葉で説明できるレベルの解説アーティファクトを作る。既存ドシエとシリーズで統一感を持たせる。

## 手順

1. **対象の特定**: 引数から対象機能と差分範囲を確定する。
   - PR番号 → `gh pr view <n> --json title,files,commits`
   - ブランチ/機能名 → `git log origin/main --oneline` から該当コミットを特定
   - リンクに使うコミットハッシュは必ず origin/main 上に存在するものだけを使う（ローカルにしかないハッシュはリンク切れになる）
2. **実装調査**: 差分に含まれるモデル・コントローラ・ビュー・マイグレーション・テストを**実ファイルの原文で**読む。コード抜粋を要約や記憶から書かない（1文字でも実コードと違うと、レビュー会で口頭説明する際の信頼を損なう）。
3. **フォーマット読込**: `${CLAUDE_SKILL_DIR}/reference/dossier-format.md` を読み、セクション構成・デザイン仕様・リンク規約に従う。スタイルの正典は `${CLAUDE_SKILL_DIR}/reference/example.html`。
4. **生成**: artifact-design スキルをロードした上で、スクラッチパッドに `dossier-<機能名英語ケバブ>.html` を書き、Artifact ツールで公開する（favicon: 📚 で統一。新機能 = 新ファイル名 = 新URL）。
5. **報告**: アーティファクトURL・その機能の技術的ハイライト・想定問答に含めた弱点の要点を最終メッセージで伝える。

## 原則

- 「なぜそうするのか」に加えて「そうしないとどうなるか」を必ず書く（CLAUDE.md の回答スタイルと同じ）
- 各セクションに Rails ガイド該当章（railsguides.jp）のURL・なぜその章かの一言・英語検索キーワードを添える
- 弱点・未実装・既知の乖離は隠さず、「聞かれたらこう答える」回答例をつける
- スキーマ変更を含む機能は ER 図（インラインSVG・テーマ追従）で該当部分を図示する
