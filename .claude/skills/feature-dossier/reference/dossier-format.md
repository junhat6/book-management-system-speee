# 実装解説ドシエ フォーマット仕様

シリーズものとして既存ドシエと見た目・構成を揃える。スタイルの正典は同ディレクトリの `example.html`（必須要件版ドシエの実物）。CSS カスタムプロパティ・コンポーネント class・Ruby ミニハイライタ JS はそこから**そのまま流用**してよい。

## ページ構成（上から順に）

1. **目次サイドバー** — sticky、IntersectionObserver によるスクロールスパイ、940px 以下では非表示
2. **ヘッダー** — eyebrow（`レビュー会 準備資料 · YYYY-MM-DD`）／明朝体タイトル（`.serif`）／1〜2文のサブ説明／スタック chips／リポジトリ・テストコマンド行
3. **機能セクション**（下記フォーマット）
4. **想定問答セクション**（最後に必ず）
5. **フッター** — 生成日と main の HEAD ハッシュ

## 機能セクションの共通フォーマット（順序固定）

| # | ブロック | class | 内容 |
|---|---|---|---|
| ① | 何を実装したか | `.lead` | 1〜2文。`<strong>` で技術的な要点を強調 |
| ② | コード抜粋＋解説 | `.codebox` | ファイルパスバー付き。**実ファイルの原文のみ**。1ブロック25行以内を目安に要点を抜粋し、行内コメントで補足してよい |
| ③ | 活用した Rails 機能 | `.feats` | グリッドカード。機能名＋バージョンタグ（`.vtag`: 7.1+ / 8.0+ / 8.1 等）＋1行説明 |
| ④ | なぜこの設計か | `.why` | コールアウト。**「そうしないとどうなるか」を必ず含める**。テストの見どころがあれば `.note` を追加 |
| ⑤ | 差分を見る | `.difflist` | PR があれば `/pull/<n>/files` を先頭に、コミットは時系列で。1行説明つき |
| ⑥ | 情報源 | `.sources` | railsguides.jp の該当章＋なぜその章かの一言＋英語検索キーワード（`code` タグ） |

## 想定問答セクション

- 冒頭に**強み3つ**（`.strengths` カード：太字見出し＋1行説明＋該当セクション番号への参照）
- Q&A は `<details class="qa">`。verdict バッジ：強み = `.verdict.ok`、未実装・改善余地 = `.verdict.gap`
- **弱点には必ず「やるならこうする」の具体的な実装方針を添える**（弱点を設計理解の証拠に変えるのが目的）

## デザイン原則

- アクセントはルビーレッド（light: `--accent: #A31631` / dark: `#E36A80`）。`example.html` の `:root` トークン一式を転用する
- 見出しは明朝（`.serif` = Hiragino Mincho ProN 系）、本文はゴシック、コードは monospace。Web フォントは使わない（システムフォントのみ）
- テーマは三段構え：`:root` のトークン定義 → `@media (prefers-color-scheme: dark)` で再定義 → `:root[data-theme="dark"]` / `:root[data-theme="light"]` で再々定義（ビューアのトグルが最優先で効く）
- CSP 制約：外部リソース（CDN・フォント・画像 URL・fetch）は一切不可。すべてインライン
- 横スクロールは各コンテナ内のみ（`.codebox pre` / `.tblwrap` / `.er-wrap`）。ページ本体は横スクロールさせない
- コードハイライトは `example.html` 末尾のミニハイライタ JS（`pre > code.ruby` を1パス走査）を流用
- ER 図はインライン SVG。色は CSS 変数参照（`var(--accent)` 等）でテーマ追従させる。`overflow-x: auto` のラッパーに入れ `min-width` を指定

## リンク規約

- diff リンク: `https://github.com/junhat6/book-management-system-speee/commit/<hash>` または `/pull/<n>/files`
- 公開前に `git log origin/main --oneline` でハッシュの実在を確認（ローカル限定ハッシュはリンク切れ）
- アーティファクトのファイル名: `dossier-<機能名英語ケバブ>.html`。新機能は新ファイル（新URL）、同一機能の更新は同名ファイルで再公開（同URL維持）
