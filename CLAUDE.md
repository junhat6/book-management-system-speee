# 進め方

- TDD(テスト駆動開発)で進める
  - テストを書いたら実装前に一度実行し、**意図した理由・意図した箇所でRed（失敗）になっているか**を出力メッセージで目視確認してから実装に進む
  - 実装が無くても通ってしまう、または別の理由で失敗しているテストは偽陽性であり、将来のバグを見逃す温床になる。「Redになった」だけで満足せず、失敗理由が期待通りかまで確認する

## テスト実行コマンド

- `rails test`

## 情報源の提示ルール

回答には必ず以下をセットで添えること：

1. **Rails ガイドの該当章**
   - 章のタイトルと URL（`https://guides.rubyonrails.org/...`）
   - **なぜその章か**の理由を1行で添える

2. **検索キーワードの提案**
   - 英語キーワードを推奨（公式ドキュメントへのアクセスが増える）
   - 例：`Rails has_many through association`

## 回答スタイル

- 情報源（ガイドのURL + 検索キーワード）を提示する
- 「なぜそうするのか」の背景・設計上の理由を必ず添える
- 「そうしないとどうなるか」も可能な限り触れる

## 参照する情報源（優先順）

1. Rails ガイド（日本語）: `https://railsguides.jp/`
2. Rails ガイド（英語原文）: `https://guides.rubyonrails.org/`
3. Rails API ドキュメント: `https://api.rubyonrails.org/`
