# 設計判断記録：著者機能（書籍×著者の多対多）

> このドキュメントは「要件をどう実現するか」について下した**設計判断とその理由**を記録するもの。
> `requirements/`（＝与えられた要件・動かせない入力）とは性質が異なるため `docs/` に分離している。
> フォーマットは ADR（Architecture Decision Record）の考え方に倣う：**決定 / 背景 / 検討した代替案 / 理由**。

- 対象機能: 書籍（多）― 著者（多）の多対多リレーション
- 関連要件: `requirements/requirements.md`（リレーション構造）、`requirements/er_diagram.md`、`requirements/screen_design.md`（書籍登録フォーム）
- 記録日: 2026-06-28

---

## 全体構造

```
books ||--|{ book_authors }|--|| authors
```

`books` と `authors` は多対多。RDBでは多対多を直接表現できないため、「どの本とどの著者が結びつくか」を 1 行 = 1 ペアで持つ**中間テーブル `book_authors`** を挟む。

---

## ① アソシエーションの張り方

**決定: `has_many :through`（中間モデル `BookAuthor` を持つ）**

- **代替案**: `has_and_belongs_to_many`（中間モデルなし・テーブルのみ）
- **理由**:
  - 中間テーブルに将来「役割（著 / 訳 / 監修）」などの属性が乗りうる。`through` なら中間モデルに属性を足せるが、`habtm` は中間モデルが無く作り直しになる。
  - ER 図が `book_authors` を `id` / `timestamps` 付きの独立テーブルとして描いており、これは `through` 前提の設計。
- **そうしないとどうなるか**: `habtm` を選ぶと、後から中間に属性が必要になった時点でテーブル・関連の再設計が発生する。

---

## ② 中間テーブル `book_authors` の制約設計

### ②-a 主キー

**決定: 代理キー（surrogate key）`id` を採用**

- **代替案**: `(book_id, author_id)` の複合主キー（natural key）
- **検討メモ**: 複合主キーでも二重登録は防げる（主キーは定義上ユニーク）。Rails も 7.1+ で複合主キーを正式サポートしており、本プロジェクトの Rails 8.1 なら機能的には使える。
- **代理キーを選んだ理由**:
  1. Rails の規約・エコシステム（gem・管理画面・各種ヘルパー）はすべて単一 `id` を前提にしており無摩擦。
  2. 主キー不変性 — `id` は意味を持たないので永遠に変わらない。`author_id` を主キーにすると「著者の付け替え」で参照関係が壊れうる。
  3. 1 行を単独で指せる（`find(id)` 一発）。将来 `book_authors` に属性が乗った時の編集・削除や `dependent` 処理が素直。
- **コスト**: `id` カラム 1 つ + 後述のユニークインデックス 1 つ。これで将来の自由度を買う。
- **補足**: 複合主キーは「自然キー vs 代理キー」というDB設計の古典的論点。今回は変化への強さを優先して代理キーに倒した。

### ②-b 重複防止

**決定: 複合ユニークインデックス `[book_id, author_id] unique` を張る**

- **理由**: アプリ層のバリデーションは並行リクエストですり抜ける。DB のユニーク制約が最終防衛線。
- アプリ層（`uniqueness: scope:`）= UX 向け、DB のユニークインデックス = 整合性の最終保証、の二層で守る。

### ②-c 外部キー制約

**決定: `book_id` / `author_id` に `foreign_key: true`**

- **理由**: 存在しない本・著者を指す中間行を DB が拒否。整合性をアプリだけでなく DB にも守らせる。

### ②-d NULL 制約

**決定: FK 両方に `null: false`**

- **理由**: 親が無い中間行は意味を持たない。「片方だけ紐づく」状態を構造的に排除する。

---

## ③ 削除時の挙動（`dependent`）

**決定: 方向ごとに非対称**

| 方向 | 設定 | 理由 |
|---|---|---|
| Book → book_authors | `dependent: :destroy`（道連れ） | 本が消えれば「その本と著者の対応」も無意味。残すと孤児レコードになる。書籍削除は画面設計にある正規操作。 |
| Author → book_authors | `dependent: :restrict_with_error`（紐づきがあれば拒否） | 画面設計に著者削除画面が無い。著者は書籍フォーム経由でしか生まれない想定。うっかり削除で本の著者欄が空になる事故を構造的に防ぐ。 |

- **考え方**: 「親子のどちらを“主”とみなすか」で向きが変わる。本→中間は「本が主」だから道連れ、著者→中間は中間の方が重い（貸出実績等の土台）から守る。

---

## ④ フォームでの著者入力方式

**決定: 既存選択（複数選択）＋ 新規追加（テキスト入力）のハイブリッド最小構成**

- **代替案**:
  - 既存から複数選択のみ（`collection_select`）— 新規追加要件を満たせない
  - `accepts_nested_attributes_for`（ネストフォーム）— 学習コスト高め、今回は過剰
- **理由**: 要件（`screen_design.md`）の「既存著者選択 ＋ 新規追加」を最小手数で両立できる。
- **構成**:
  - 既存著者 → 複数選択。`has_many :through` が自動生成する `author_ids=` setter に配列で渡す。
  - 新規著者 → テキスト入力。`books` テーブルに無い「仮想の入力」として受け取り、保存時に著者として取り込む。

- **補足（Insight）**: `has_many :through` を張ると `book.author_ids = [1,3,5]` という setter が自動生成され、配列を渡すだけで中間テーブルの差分 INSERT/DELETE を Rails が同期する。既存選択がラクなのはこの仕組みのおかげ。

---

## ⑤ Strong Parameters

**決定: 2 種類を受け取る**

- 既存選択 → `author_ids: []`（**配列許可の `[]` が必須**。これが無いと弾かれる）
- 新規テキスト → 実カラムではない仮想入力。**この取り込み方が本機能の主要な設計判断（実装時の `TODO(human)`）**。
  - 選択肢A: 仮想属性（`attr_accessor`）でモデルに一時保持しコールバックで取り込む → モデルに責務が集まり再利用しやすい
  - 選択肢B: コントローラで直接さばく → 手軽だがコントローラが太る
  - ※この A/B の判断は実装時に確定させる。

## ⑥ 著者名の一意性・重複登録の扱い

**決定: `authors.name` にユニーク制約 ＋ 新規追加時は `find_or_create_by(name:)`**

- **理由**:
  - テキスト入力で同名著者（例「夏目漱石」）を毎回新規作成すると重複が量産される。`find_or_create_by` で「あれば再利用、無ければ作成」。
  - DB 側の `name` ユニークインデックスで並行リクエストの二重作成も防ぐ（②と同じ二層思想）。
- **割り切り**: 同姓同名の別人問題は今回スコープ外。学習課題としては「名前で一意」で十分。厳密化すると著者プロフィール等の別設計が必要になり課題の焦点がぼやける。

## 実装の順番（DB層 → モデル → フォーム）

1. `Author` モデル + マイグレーション（`name` + ユニークインデックス）
2. `BookAuthor` 中間モデル + マイグレーション（②の制約フル装備）
3. 3 モデルのアソシエーション記述（①③）
4. 書籍フォーム + Strong Params + 新規著者の取り込み（④⑤⑥）← `TODO(human)` を置く箇所
